# Visio Flowchart

業務フローや処理手順を、Codex で扱いやすい Mermaid ソースと出力手順に変換する。

## ワークフロー

### 1. 入力を整理する

- ユーザーが自然言語で説明した場合は、登場人物、処理順、条件分岐、ループ、例外系を抽出する。
- ユーザーが既存の Mermaid を渡した場合は、そのままレビューに進む。
- 図の目的が曖昧なら、共有用なのか実装検討用なのかを文脈から判断してラベル粒度を調整する。

### 2. Mermaid を組み立てる

- 参加者やシステム間の受け渡しが中心なら `sequenceDiagram` を使う。
- 手順、条件分岐、再試行の流れが中心なら `flowchart TD` を使う。
- スイムレーン風に見せたい場合は、`subgraph` か `participant` を使って責務境界を明示する。
- 分岐は `alt / else / end` または判定ノードで表現する。
- 自己処理や内部処理は明示的に書く。

### 3. 出力を整える

- まず Mermaid コードを返す。
- その後に、図の読み方、曖昧な点、必要なら改善案を短く添える。
- 参考例が必要なら、このリポジトリの [docs/site-structure-mermaid.md](/Users/ryukouokumura/Desktop/boss-workspace/ai-seitai-personal-media/docs/site-structure-mermaid.md) を確認する。

### 4. 画像化が必要な場合に対応する

- このリポジトリには現在 `scripts/render_flowchart.py` のような専用レンダラーが同梱されていない前提で進める。
- 画像生成を求められたら、まず利用可能なレンダラーや CLI があるか確認する。
- レンダラーがある場合は、作業ファイルを `tmp/flowchart/`、成果物を `output/flowchart/` に置く。
- レンダラーがない場合は、Mermaid ソースを完成版として返し、不足しているのが画像レンダリング手段だけであると明示する。

## 設計ルール

- ノード名は短く保つ。
- 1 ノード 1 アクションを基本にする。
- 条件文は「承認済み?」「入力あり?」のように Yes/No で読める形にする。
- 実装詳細よりも業務の意思決定点を優先して図に載せる。
- 色や装飾が必要でも、まずは構造が正しい Mermaid を優先する。

## 出力テンプレート

1. 前提や補足を 1 から 3 行で整理する
2. Mermaid コードブロックを出す
3. 必要なら「画像化するには何が足りないか」を 1 段落で伝える
