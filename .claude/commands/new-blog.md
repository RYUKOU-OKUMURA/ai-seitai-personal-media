ブログ記事を新規作成するスキル。

ユーザーから提供された情報: $ARGUMENTS

## 手順

1. ユーザーの入力からブログ記事のテーマ・内容を把握する
2. ファイル名を決定する（日本語タイトルをハイフン区切りにする。例: `仕組み化の本質-悩まず決めることが重要.md`）
3. 以下のフォーマットに従って `src/content/blog/` に新しい `.md` ファイルを作成する

## フロントマターのフォーマット

```yaml
---
title: "メインテーマ——サブメッセージ"
publishedAt: "YYYY-MM-DDT00:00:00.000Z"
category: "カテゴリ名"
image: "/public/images/blog/ファイル名"
excerpt: "要約文（1〜2文）"
draft: false
---
```

### 各フィールドの説明

| フィールド | 必須 | 説明 |
|---|---|---|
| `title` | 必須 | 「メインテーマ——サブメッセージ」の形式。ダッシュ（——）で区切る。具体的で読者の興味を引く表現にする |
| `publishedAt` | 必須 | 公開日。ISO 8601形式（例: `"2026-02-11T00:00:00.000Z"`）。特に指定がなければ今日の日付を使用 |
| `category` | 必須 | カテゴリ。既存例: "マインドセット", "チームビルディング", "実践ノウハウ" |
| `image` | 任意 | アイキャッチ画像パス（`/public/images/blog/` 配下）。指定がなければ空文字 |
| `excerpt` | 必須 | 記事の要約。1〜2文で核心となるメッセージを端的に伝える |
| `draft` | 任意 | 下書きにしたい場合は `true`。デフォルトは `false` |

## 本文の構成ルール

### 全体構成（この順番を守る）

1. **導入段落** — 「結論から言うと」で始め、テーマの核心を先に提示する。その後「今日は〜について正直に書いてみます」のように記事の目的を宣言する
2. **H2セクション × 2〜3本** — 問題提起 → 定義・フレームワーク → 補足・注意点の流れで展開する
3. **「今日のまとめ」セクション（H2）** — 番号付きリストで要点を整理する。各項目は **太字のキーメッセージ** + ダッシュ（——）+ 補足説明の形式
4. **締めの一言** — 「でわでわ、次回の記事で！」で終わる

### 文体・トーンのルール

- 一人称は「僕」を使う
- 語尾は「〜なんですよね」「〜なんです」「〜なんですよ」を多用し、口語的で親しみやすいトーンにする
- 「正直に言うと」「これまで僕も〜だと思っていました。でも、それは間違いだったんです」のように、正直さ・率直さを出す
- 整体師・店舗ビジネスの現場視点から語る
- 「BOSS定義の〜」のように、自分独自の定義・フレームワークを提示する場合はそれを明示する
- 敬語と口語のミックス（ですます調ベースだが「〜だよね」「〜じゃなくて」なども混ぜる）

### 強調・装飾のルール

- 最も伝えたいキーワード・フレーズは `**太字**` にする
- 太字は1セクションにつき1〜2箇所に絞る（多用しない）
- まとめリストでは各項目の冒頭メッセージを太字にする

### 文章の書き方

- 1段落は2〜4文程度。長くなりすぎない
- 具体例を挟む（「例えば〜」）
- 「〜けど、〜」「〜なんだけど、〜」で逆接を自然に繋げる
- 読者への問いかけや共感を含める（「これはよく言われることですよね」など）

## ファイル名のルール

- 日本語のタイトルからキーワードを抽出し、ハイフン区切りにする
- 例: タイトル「仕組み化の本質——悩まず決めることが重要」→ ファイル名 `仕組み化の本質-悩まず決めることが重要.md`
- ダッシュ（——）はハイフン（-）に置き換える

## 注意事項

- ユーザーが指定していない必須フィールドがあれば、作成前に確認する
- `image` が指定されていない場合は空文字 `""` にする
- ユーザーからはテーマやキーワード程度の指示でも、上記の構成・文体ルールに従って完成度の高い記事を生成する
